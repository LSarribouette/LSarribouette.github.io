Les tests unitaires avec JUnit

JUnit est un framework de tests unitaires pour Java, notamment intégré par défaut dans les IDE Eclipse et IntelliJ.

Un test unitaire se construit en 3 parties (AAA), après avoir déterminer le scénario à tester :

  • Arrange ~ initialisation des données et de l’état des objets
  • Act ~ exécuter le traitement
  • Assert ~ comparer le résultat obtenu avec le résultat attendu et vérifier l’état des objets
Quelques principes : ”un test a un objectif unique”, ”un test est le plus petit et simple possible”, ”chaque test est isolé et ne dépend pas d’aucun autre test”, ”les tests sont exécutés régulièrement (ce qui amène à l’automatisation)”.

Annotations JUnit

JUnit utilise des annotations pour diriger l’exécution des tests.

  • @Test pour définir les méthodes de test
  • @BeforeAll pour déclencher une méthode avant l’exécution du premier test de la classe (par exemple pour initialiser des ressources partagées par plusieurs tests)
  • AfterAll pour déclencher une méthode après l’exécution du dernier test de la classe (par exemple pour fermer des ressources partagées par plusieurs tests)
  • @BeforeEach et @AfterEach pour déclencher une méthode avant ou après l’exécution de chaque test de la classe
  • @RunWith et @SuiteClasses pour définir une campagne de test

Méthodologie

On importe les packages issus de org.junit.*, qui donne accès à des classes comme “org.junit.jupiter.api.Assertions”.

BP : il est recommandé de les importer en static : import static org.junit.jupiter.api.Assertions.assertEquals;

On crée une classe pour effectuer un (et un seul) test.

On lui ajoute une suite de méthodes pour effectuer le traitement, à l’aide des annotations disponibles.

L’utilisation de la classe “org.junit.jupiter.api.Assertions” donne accès à un ensemble de méthodes permettant de tester si l’assertion testée est vérifiée (le test passe au vert = réussite) ou non (le test passe au rouge = échec).

La méthode “assertThrows” permet de vérifier la levée d’une exception.

On peut créer une campagne de tests avec l’annotation @Suite, suivie des différentes classes (@SelectClasses) et packages (@SelectPackages) de tests à exécuter.

Couverture des tests

La couverture de tests permet de mesurer le pourcentage de code vérifié par les tests et d’identifier les lignes de code testées et non testées.

Notion de bouchon

Dans le cas d’un test unitaire, un seul composant est testé indépendamment des autres. Néanmoins, une méthode en appelle souvent d’autres, situées potentiellement dans des classes différentes, qui ont elles-mêmes leurs dépendances.

La notion de bouchon (stub) permet de simuler le comportement des dépendances en fonction des scénarios du test.

On dit alors qu’on mock -on reproduit le comportement- de certaines dépendances.
